# 多使用者短暫工作階段 (Multi-User Short-Lived Sessions)

示範一種 **無狀態後端模式**，其中多個使用者透過 **暫時性工作階段** 與共享的 `copilot` 伺服器互動。工作階段會針對每個請求建立並銷毀，並透過個別使用者的虛擬檔案系統實現隔離。

## 架構

```
┌──────────────────────┐
│    Copilot CLI       │  (無介面 TCP 伺服器)
│    (共享伺服器)       │
└───┬──────┬───────┬───┘
    │      │       │   基於 TCP 的 JSON-RPC (cliUrl)
    │      │       │
┌───┴──┐ ┌┴────┐ ┌┴─────┐
│ C1   │ │ C2  │ │  C3  │
│使用者A│ │使用者A│ │使用者B│
│(新建) │ │(新建) │ │(新建) │
└──────┘ └─────┘ └──────┘

每個請求 → 新工作階段 → 回應後中斷連線
每個使用者一個虛擬檔案系統 (記憶體內，不跨使用者共享)
```

## 示範內容

1. **暫時性工作階段** — 每次互動都會建立一個全新的工作階段，並在完成後立即銷毀。伺服器端不會在請求之間保留任何狀態。
2. **每使用者虛擬檔案系統** — 自訂工具 (`write_file`, `read_file`, `list_files`) 由記憶體中的 Map 支援。每個使用者都有自己的隔離檔案系統執行個體 —— 使用者 A 的檔案對使用者 B 是不可見的。
3. **應用程式層狀態** — 雖然工作階段是無狀態的，但應用程式會在同一使用者的請求之間維護狀態 (虛擬檔案系統)。這反映了真實的後端場景，其中工作階段狀態存在於您的資料庫中，而非 LLM 工作階段中。
4. **自訂工具** — 使用 `defineTool` 搭配 `availableTools: []` 來用受控的虛擬檔案系統取代所有內建工具。
5. **多用戶端隔離** — 使用者 A 的兩個用戶端共享相同的虛擬檔案系統 (同一使用者)，但使用者 B 的虛擬檔案系統是完全分開的。

## 每個用戶端的操作內容

| 用戶端 | 使用者 | 動作 |
|--------|------|--------|
| **C1** | A | 在使用者 A 的虛擬檔案系統中建立 `notes.md` |
| **C2** | A | 列出檔案並讀取 `notes.md` (可以看到 C1 的檔案，因為是同一使用者的檔案系統) |
| **C3** | B | 列出使用者 B 虛擬檔案系統中的檔案 (為空 —— 完全隔離) |

## 設定

| 選項 | 值 |
|--------|-------|
| `cliUrl` | 共享伺服器 |
| `availableTools` | `[]` (無內建工具) |
| `tools` | `[write_file, read_file, list_files]` (每使用者虛擬檔案系統) |
| `sessionId` | 自動產生 (暫時性) |

## 何時使用此模式

- **API 後端** — 無工作階段持久化的無狀態請求/回應
- **無伺服器函式 (Serverless functions)** — 每次調用都是獨立的
- **高吞吐量服務** — 請求之間無工作階段開銷
- **隱私敏感型應用程式** — 對話歷程記錄永不持久化

## 執行

```bash
./verify.sh
```

需要 `copilot` 二進位檔案 (自動偵測或設定 `COPILOT_CLI_PATH`) 和 `GITHUB_TOKEN`。

# 多使用者長期工作階段 (Multi-User Long-Lived Sessions)

示範一種 **類似生產環境的多使用者設定**，其中多個用戶端透過儲存在磁碟上的 **持久、長期的工作階段** 共享單個 `copilot` 伺服器。

## 架構

```
┌──────────────────────┐
│    Copilot CLI       │  (無介面 TCP 伺服器)
│    (共享伺服器)       │
└───┬──────┬───────┬───┘
    │      │       │   基於 TCP 的 JSON-RPC (cliUrl)
    │      │       │
┌───┴──┐ ┌┴────┐ ┌┴─────┐
│ C1   │ │ C2  │ │  C3  │
│使用者A│ │使用者A│ │使用者B│
│工作階段1│ │工作階段1│ │工作階段2│
│      │ │(恢復)  │     │
└──────┘ └─────┘ └──────┘
```

## 示範內容

1. **共享伺服器** — 單個 `copilot` 執行個體透過 TCP 為多個使用者和工作階段提供服務。
2. **每使用者配置隔離** — 每個使用者在磁碟上都有自己的 `configDir` (`tmp/user-a/`, `tmp/user-b/`)，因此配置、記錄和狀態完全分開。
3. **跨用戶端工作階段共享** — 使用者 A 的用戶端 1 建立一個工作階段並教導它一個事實。用戶端 2 恢復同一個工作階段 (透過 `sessionId`) 並檢索該事實 —— 示範了跨用戶端的工作階段連續性。
4. **使用者之間的工作階段隔離** — 使用者 B 在完全獨立的工作階段中操作，無法看到使用者 A 的對話歷程記錄。
5. **磁碟持久化** — 工作階段狀態被寫入真實的 `tmp/` 目錄，模擬生產環境的持久化 (在執行後清理)。

## 每個用戶端的操作內容

| 用戶端 | 使用者 | 動作 |
|--------|------|--------|
| **C1** | A | 建立工作階段 `user-a-project-session`，教導它一個代號 |
| **C2** | A | 恢復 `user-a-project-session`，確認它記得該代號 |
| **C3** | B | 建立獨立的工作階段 `user-b-solo-session`，驗證它不知道使用者 A 的資料 |

## 設定

| 選項 | 使用者 A | 使用者 B |
|--------|--------|--------|
| `cliUrl` | 共享伺服器 | 共享伺服器 |
| `configDir` | `tmp/user-a/` | `tmp/user-b/` |
| `sessionId` | `user-a-project-session` | `user-b-solo-session` |
| `availableTools` | `[]` | `[]` |

## 何時使用此模式

- **SaaS 平台** — 每個租戶獲得隔離的配置和持久工作階段
- **團隊協作工具** — 多個團隊成員在同一個專案上共享工作階段
- **IDE 後端** — 使用者在多個編輯器/分頁中開啟同一個專案

## 執行

```bash
./verify.sh
```

需要 `copilot` 二進位檔案 (自動偵測或設定 `COPILOT_CLI_PATH`) 和 `GITHUB_TOKEN`。

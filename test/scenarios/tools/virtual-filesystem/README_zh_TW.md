# 設定範例：虛擬檔案系統

展示如何使用 **由記憶體儲存端支援的自定義工具實作**（而非真實檔案系統）來執行 Copilot 代理程式。代理程式並不知道它是虛擬的 — 它看到的 `create_file`、`read_file` 和 `list_files` 工具運作正常，但沒有任何位元組會觸及磁碟。

此模式是以下各項的基礎：
- **WASM / 瀏覽器代理程式**，其中沒有真實的檔案系統
- **雲端託管的沙盒**，其中檔案操作會進入物件儲存空間
- **多租戶平台**，其中每個使用者都獲得隔離的虛擬儲存空間
- **Office 增益集**，其中「檔案」是記憶體中的文件區塊

## 運作方式

1. **停用所有內建工具**，使用 `availableTools: []`
2. **提供自定義工具** (`create_file`、`read_file`、`list_files`)，其處理常式在主機程序中讀取/寫入 `Map` / `dict` / `HashMap`
3. **自動核准權限** — 由於工具完全由使用者控制，因此不需要對話方塊
4. 代理程式正常使用工具 — 它不知道它們是虛擬的

## 每個範例的功能

1. 建立一個不含內建工具 + 3 個自定義虛擬檔案系統工具的工作階段
2. 發送：_"建立一個名為 plan.md 的檔案，其中包含建置 CLI 工具的簡短 3 項目專案計畫。然後讀回它並告訴我你寫了什麼。"_
3. 代理程式呼叫 `create_file` → 寫入記憶體映射
4. 代理程式呼叫 `read_file` → 從記憶體映射讀取
5. 印出代理程式的回應
6. 傾印記憶體儲存內容，以證明檔案僅存在於記憶體中

## 設定

| 選項 | 值 | 效果 |
|--------|-------|--------|
| `availableTools` | `[]` (空) | 移除所有內建工具 (bash, view, edit, create_file, grep, glob 等) |
| `tools` | `[create_file, read_file, list_files]` | 由記憶體儲存端支援的自定義工具 |
| `onPermissionRequest` | 自動核准 | 無權限對話方塊 |
| `hooks.onPreToolUse` | 自動允許 | 無工具確認提示 |

## 關鍵見解

整合者控制工具層。藉由將內建工具替換為自定義實作，您可以將支援的儲存端切換為任何內容 — `Map`、Redis、S3、SQLite、IndexedDB — 而代理程式既不知道也不在乎。系統提示保持不變。代理程式正常進行計畫和操作。

與內建工具同名的自定義工具會自動覆蓋內建工具 — 無需明確排除它們。`availableTools: []` 會移除所有內建工具，同時保留您的自定義工具。

## 執行

```bash
./verify.sh
```

需要 `copilot` 二進位檔 (自動偵測或設定 `COPILOT_CLI_PATH`) 和 `GITHUB_TOKEN`。

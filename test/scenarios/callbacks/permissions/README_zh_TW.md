# 配置範例：權限 (Permissions)

演示 **權限請求流程** — 執行階段在執行工具之前會向 SDK 請求權限，而 SDK 可以核准或拒絕每個請求。此範例核准所有請求，同時記錄調用了哪些工具。

此模式是以下功能的基礎：
- **企業策略強制執行**：限制某些工具的使用
- **稽核記錄 (Audit logging)**：必須記錄所有工具調用
- **互動式核准 UI**：由人員確認敏感操作
- **細粒度存取控制**：根據工具名稱、參數或上下文進行控制

## 運作方式

1. 在工作階段配置上 **啟用 `onPermissionRequest` 處理常式**
2. 在記錄陣列中 **追蹤哪些工具請求了權限**
3. **核准所有權限請求** (回傳 `kind: "approved"`)
4. **傳送觸發工具使用的提示** (例如：透過 glob 列出檔案)
5. **列印權限記錄**，顯示哪些工具已獲核准

## 每個範例的操作內容

1. 建立一個具有 `onPermissionRequest` 回呼 (callback) 的工作階段，該回呼會記錄並核准請求
2. 傳送：_"使用 glob 並以模式 '*' 列出目前目錄中的檔案。"_
3. 執行階段在每次工具執行前呼叫 `onPermissionRequest`
4. 回呼記錄 `approved:<toolName>` 並回傳核准
5. 列印代理程式的請求
6. 傾印 (dump) 權限記錄，顯示所有獲核准的工具調用

## 配置

| 選項 | 值 | 效果 |
|--------|-------|--------|
| `onPermissionRequest` | 記錄 + 核准 | 記錄工具名稱，回傳 `approved` |
| `hooks.onPreToolUse` | 自動允許 | 無工具確認提示 |

## 關鍵洞察

`onPermissionRequest` 處理常式讓整合者能夠完全控制代理程式可以執行的工具。透過檢查請求 (工具名稱、參數)，您可以實作允許/拒絕清單、對危險操作要求人工核准，或記錄每個操作以符合合規性。回傳 `{ kind: "denied" }` 會阻止工具執行。

## 執行

```bash
./verify.sh
```

需要 `copilot` 二進位檔案 (自動偵測或設定 `COPILOT_CLI_PATH`) 和 `GITHUB_TOKEN`。
